
# 追漫阁项目优化计划

## 1. 文档目的

本文档基于 `PROJECT_ANALYSIS.md` 对追漫阁项目进行系统性优化规划，目标是将当前“可用的个人追更工具”提升为“稳定、安全、可维护、可扩展、可观测的自部署追更平台”。

本计划覆盖以下方面：

- 优化目标
- 优化范围
- 优先级排序
- 实施策略
- 资源分配
- 阶段排期
- 风险评估
- 成功指标
- 验收标准
- 责任角色划分

## 2. 战略目标

### 2.1 总体目标

在保持项目轻量、自部署、低维护成本优势的前提下，优先解决安全、稳定性、可维护性和同步可靠性问题，再逐步增强扩展能力、用户体验和运维可观测性。

### 2.2 分项目标

| 优化方向 | 当前状态 | 目标状态 | 优先级 |
| --- | --- | --- | --- |
| 安全性 | 已有登录、CSRF、bcrypt、限流，但 CSP 和容器权限仍可加强 | 达到个人公网部署的基础安全标准 | P0 |
| 稳定性 | 依赖 TMDB、Invidious、Telegram，外部服务波动会影响体验 | 具备重试、降级、健康评分和失败恢复能力 | P0 |
| 可维护性 | 核心文件职责偏大，如 routes.py、database.py、app.js | 按领域拆分模块，降低单文件复杂度 | P1 |
| 性能 | 已有缓存、WAL、连接池和并发同步 | 降低重复外部请求，提升同步效率和页面响应 | P1 |
| 可观测性 | 已有 Prometheus 指标和同步日志 | 增加关键业务指标、错误分类和运维看板基础 | P1 |
| 可扩展性 | 当前适合单用户和 SQLite 场景 | 为多视频源、多用户、独立 worker 预留架构边界 | P2 |
| 用户体验 | 已有主题、Toast、SSE 进度 | 增强同步可解释性、失败提示、批量操作和配置体验 | P2 |
| 测试质量 | 项目已有部分测试基础，但覆盖面仍需提升 | 核心算法、数据库、API 和同步流程具备自动化测试 | P1 |

## 3. 优化原则

1. 先治理风险，再增强功能。
2. 先保护数据，再改造结构。
3. 先补测试，再做重构。
4. 保持轻量部署，不盲目引入复杂基础设施。
5. 所有优化项必须可衡量、可验收、可回滚。
6. 优先使用项目已有工具、模块和技术栈。
7. 优化过程不得破坏现有 Docker 部署方式。
8. 任何数据库结构调整必须通过迁移脚本完成。

## 4. 当前痛点归纳

### 4.1 安全痛点

| 痛点 | 影响 | 建议等级 |
| --- | --- | --- |
| docker-compose 覆盖镜像非 root 用户，以 root 身份运行容器 | 容器逃逸或文件误写风险增加 | P0 |
| CSP 允许 `unsafe-inline` | XSS 防护强度下降 | P1 |
| 单密码认证模式缺少审计能力 | 无法区分不同用户或追踪操作来源 | P2 |
| 生产环境 SECRET_KEY 可能未显式配置 | 重启导致会话失效，也影响安全稳定性 | P0 |
| 外部 Token 配置依赖环境和 settings，缺少统一敏感配置检查 | 易出现误提交或配置缺失 | P1 |

### 4.2 稳定性痛点

| 痛点 | 影响 | 建议等级 |
| --- | --- | --- |
| 自部署 Invidious 实例可用性依赖本机服务、网络出口和 YouTube 上游响应 | 视频源搜索失败或响应慢 | P0 |
| 外部 API 请求缺少统一退避策略 | 短时故障时容易连续失败 | P0 |
| 调度器运行在 Web 进程内 | 横向扩容时可能重复执行任务 | P2 |
| 同步任务失败分类不够细 | 难以定位失败原因 | P1 |
| Telegram 备份失败可能仅日志记录 | 用户可能无法及时发现备份异常 | P1 |

### 4.3 可维护性痛点

| 痛点 | 影响 | 建议等级 |
| --- | --- | --- |
| `app/api/routes.py` 职责过多 | 接口维护和测试成本上升 | P1 |
| `app/db/database.py` 包含大量领域 CRUD | 后续扩展容易冲突 | P1 |
| `app/web/static/app.js` 承担多个前端领域 | 前端变更风险集中 | P2 |
| 配置项分散在 config.py、settings、环境变量中 | 配置来源不够清晰 | P1 |

### 4.4 性能痛点

| 痛点 | 影响 | 建议等级 |
| --- | --- | --- |
| 同步时每集多关键词请求外部 API | 请求量大，容易触发限流 | P1 |
| 搜索结果缺少关键词级缓存 | 重复搜索浪费时间和外部资源 | P1 |
| SimpleCache 为进程内缓存 | 多 worker 场景缓存不共享 | P2 |
| SQLite 适合低并发但不适合多用户写入 | 后续多用户扩展受限 | P3 |

## 5. 优先级定义

| 优先级 | 定义 | 处理策略 |
| --- | --- | --- |
| P0 | 安全、数据、核心可用性风险 | 第一阶段必须完成 |
| P1 | 明显影响稳定性、可维护性、性能的问题 | 第二阶段重点完成 |
| P2 | 体验增强、架构预留、扩展能力建设 | 第三阶段规划完成 |
| P3 | 长期演进或规模化后才需要 | 保留方案，不立即实施 |

## 9. 详细实施策略

### 9.1 安全优化策略

#### 9.1.1 容器权限收敛

目标：避免应用容器默认以 root 身份运行。

实施策略：

1. 检查 Dockerfile 中 `appuser` 的 UID 和 GID。
2. 调整 docker-compose 中的 `user` 配置，避免覆盖为 `0:0`。
3. 明确 `data` 目录宿主机权限要求。
4. 验证 SQLite 数据库、日志、备份文件是否可正常读写。

验收指标：

- 容器内 `id` 命令显示非 root 用户。
- 应用可正常创建和写入数据库。
- 备份文件可正常生成。
- 健康检查接口正常。

#### 9.1.2 生产环境安全配置检查

目标：避免弱配置进入生产环境。

实施策略：

1. 启动时检查 `SECRET_KEY` 是否显式配置。
2. 检查默认密码是否仍在使用。
3. 检查 Telegram Token、TMDB API Key 是否以安全方式配置。
4. 对生产模式下缺失关键配置给出明确错误或高危日志。

验收指标：

- 缺失 SECRET_KEY 时生产模式拒绝启动或输出高危告警。
- 默认密码首次生成后要求用户尽快修改。
- 日志中不得输出完整 Token 或密钥。

#### 9.1.3 CSP 收敛

目标：逐步去除 `unsafe-inline`。

实施策略：

1. 盘点模板中的内联脚本和内联样式。
2. 将内联逻辑迁移到静态 JS 文件。
3. 通过 nonce 或 hash 机制兼容必要内联内容。
4. 分阶段收紧 CSP。

验收指标：

- CSP 策略中逐步减少 `unsafe-inline`。
- 页面功能无回归。
- 浏览器控制台无 CSP 阻断错误。

### 9.2 稳定性优化策略

#### 9.2.1 自部署 Invidious 可用性检测

目标：提升自部署 Invidious 服务在本机服务异常、网络出口异常和 YouTube 上游波动场景下的可诊断性与恢复能力。

实施策略：

1. 为自部署 Invidious 服务维护健康状态。
2. 记录连续失败次数、最近成功时间、平均响应时间和最近错误类型。
3. 请求失败时采用指数退避，避免在服务异常时持续压测自己的实例。
4. 区分本机服务不可用、网络出口异常、YouTube 上游异常和响应解析异常。
5. 定期执行轻量健康检查，并在恢复后自动解除降级状态。

建议健康状态模型：

```text
健康状态 = 服务可达性 + 平均响应时间 + 连续失败次数 + 最近错误类型 + 最近成功时间
```

验收指标：

- 自部署服务不可用时能明确记录原因。
- 服务恢复后可自动解除降级状态。
- 搜索失败率下降。
- 日志可区分本机服务异常、网络出口异常和 YouTube 上游异常。

#### 9.2.2 同步错误分类

目标：让同步失败可诊断、可统计、可恢复。

建议错误分类：

| 错误码 | 含义 | 处理方式 |
| --- | --- | --- |
| EXT_TIMEOUT | 自部署 Invidious 或外部上游请求超时 | 退避重试并记录错误来源 |
| EXT_RATE_LIMIT | 外部 API 限流 | 退避等待，降低并发 |
| EXT_BAD_RESPONSE | 外部 API 响应异常 | 记录原始状态摘要 |
| MATCH_NO_RESULT | 未找到合格视频源 | 非致命错误，继续后续集数 |
| DB_WRITE_FAILED | 数据库写入失败 | 终止当前任务并告警 |
| CONFIG_MISSING | 配置缺失 | 阻止相关功能执行 |

验收指标：

- 同步日志包含错误码。
- 前端提示能区分“未找到源”和“系统错误”。
- 自动同步任务不会因单集无源而整体失败。

### 9.3 性能优化策略

#### 9.3.1 关键词级搜索缓存

目标：减少重复请求 Invidious。

实施策略：

1. 设计搜索缓存结构，缓存关键词、实例、返回结果摘要和过期时间。
2. 对相同关键词在短时间内直接使用缓存结果。
3. 强制同步时允许绕过缓存。
4. 缓存命中率纳入监控指标。

建议缓存字段：

| 字段 | 说明 |
| --- | --- |
| keyword | 搜索关键词 |
| provider | 视频源提供方 |
| result_json | 搜索结果 JSON |
| hit_count | 命中次数 |
| expires_at | 过期时间 |
| created_at | 创建时间 |

验收指标：

- 相同关键词在缓存期内不重复请求外部 API。
- 同步耗时下降。
- 外部 API 请求量下降。

#### 9.3.2 同步策略优化

目标：减少不必要的全量同步。

实施策略：

1. 优先同步最近未观看或未找到源的集数。
2. 已完结且已看完动漫默认跳过。
3. 对长期无更新动漫降低同步频率。
4. 对近期更新动漫提高同步优先级。

验收指标：

- 单次全量同步请求数下降。
- 新集发现延迟可控。
- 用户手动强制同步仍能覆盖所有集数。

### 9.4 可维护性优化策略

#### 9.4.1 数据库访问层拆分

目标：降低 `database.py` 复杂度。

建议拆分结构：

```text
app/db/
├── connection.py
├── schema.py
├── repositories/
│   ├── anime_repository.py
│   ├── episode_repository.py
│   ├── source_repository.py
│   ├── settings_repository.py
│   ├── log_repository.py
│   └── backup_repository.py
└── database.py
```

实施策略：

1. 先补齐当前 database.py 的关键测试。
2. 保持外部函数签名兼容。
3. 分领域移动内部实现。
4. 每次拆分后运行测试。
5. 最终将 database.py 变为兼容导出层。

验收指标：

- 原有 API 调用不需要大范围修改。
- 数据库测试全部通过。
- 单个 repository 文件职责单一。

#### 9.4.2 API 蓝图拆分

目标：降低 `routes.py` 复杂度。

建议拆分结构：

```text
app/api/
├── __init__.py
├── anime_routes.py
├── source_routes.py
├── settings_routes.py
├── backup_routes.py
├── log_routes.py
└── system_routes.py
```

实施策略：

1. 保持 URL 路径不变。
2. 先抽离低耦合接口，如 health、settings、backup。
3. 再抽离 anime 和 source 相关接口。
4. 使用统一响应模块保持格式一致。

验收指标：

- 所有原 API 路径可用。
- 前端无需修改或仅少量修改。
- 路由文件职责清晰。

### 9.5 测试优化策略

目标：让重构和优化具备安全网。

测试分层建议：

| 测试类型 | 覆盖内容 | 优先级 |
| --- | --- | --- |
| 单元测试 | 文本预处理、集数提取、模糊匹配、评分 | P0 |
| 数据库测试 | CRUD、迁移、备份恢复、统计查询 | P1 |
| API 测试 | 添加动漫、进度更新、同步触发、设置保存 | P1 |
| 集成测试 | TMDB/Invidious Mock 下的完整同步流程 | P1 |
| 前端冒烟测试 | 首页、详情、设置、同步按钮 | P2 |

关键测试用例：

- 繁体标题可转换并匹配。
- 中文数字集数可正确识别。
- 合集、解说、剪辑内容被过滤。
- 相邻集数不应获得高分。
- 同一 video_id 不重复写入。
- 备份导出后可重新导入。
- 外部 API 超时时同步流程可继续处理后续集数。

## 12. 数据驱动依据

本优化计划的依据来自项目分析中的以下事实：

1. 项目核心价值依赖自部署 Invidious 服务和 YouTube 上游搜索能力，因此服务健康检测、退避重试和错误分类优先级最高。
2. 项目使用 SQLite，适合当前个人部署，但需要保护数据安全和备份恢复能力。
3. 项目已具备 Prometheus、sync_logs、backup_logs，可低成本扩展可观测指标。
4. 匹配算法是核心竞争力，因此必须优先补齐算法测试。
5. 当前 routes.py、database.py、app.js 职责集中，继续扩展会增加维护成本。
6. Dockerfile 与 docker-compose 的用户权限策略存在不一致，需要优先修正。
7. 当前安全措施已有基础，但生产配置检查和 CSP 收敛仍有提升空间。

## 15. 技术路线建议

### 15.1 保持当前主技术栈

短中期建议继续保持：

- Flask
- SQLite
- Jinja2
- 原生 JavaScript
- Docker Compose
- APScheduler

原因：

- 与个人自部署目标匹配。
- 部署成本低。
- 当前代码和功能已围绕该栈形成闭环。
- 优化重点应放在稳定性、安全和可维护性，而不是技术栈重写。

### 15.2 预留长期演进接口

长期可预留：

- 视频源 Provider 接口。
- 独立 worker 任务执行接口。
- 数据库 repository 层。
- 用户与权限模型。
- 外部缓存适配层。

这些接口可以在不改变当前部署模式的情况下逐步建设。

## 16. 最终结论

追漫阁当前已经具备完整的个人追更闭环，优化重点不应是推倒重写，而应是围绕“安全、稳定、可维护、可观测”进行渐进式增强。

最优实施路径是：

1. 第一阶段先解决容器权限、安全配置、外部 API 稳定性和错误分类。
2. 第二阶段补测试、加缓存、拆分数据库和 API 模块。
3. 第三阶段改善前端模块化和长期架构边界。
4. 第四阶段根据真实使用规模再决定是否迁移数据库、多用户化或引入独立任务队列。

按此计划执行后，项目可以在不显著增加部署复杂度的前提下，获得更强的生产可用性、更好的维护体验和更清晰的未来扩展路径。
